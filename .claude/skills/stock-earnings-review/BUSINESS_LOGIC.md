# stock-earnings-review — 业务逻辑说明（详版）

本文档保留编排层（大模型 / 规划引擎）需要遵循的**完整业务规则**，与精简后的 `SKILL.md` 配套使用。  
实现细节以 `scripts/` 下代码及仓库根目录 `api_integration.md` 为准。

调用方式说明：本 skill 采用**分步骤编排**（实体识别 -> 报告期候选 -> 模型选期 -> 业绩点评）。

---

## 1. 前置检查（进入流程前）

- 若用户问句中**未提及**任何公司名称、股票简称或股票代码，**不要**进入实体识别与后续接口调用；应直接向用户确认需要点评的公司或代码。
- 常见实体形式包括：
  - 公司全称（如「贵州茅台酒股份有限公司」）
  - 股票简称（如「东方财富」「腾讯控股」）
  - 股票代码（如「600519」「AAPL」）
- 若用户输入中**出现多个实体**，优先取**第一个**明确的公司实体。

---

## 2. 第一步：实体识别与市场校验

### 2.1 调用方式

- 调用 `scripts/validate_entity.py`，将用户的**原始问句**作为输入，请求实体识别接口，一次性完成实体识别与市场分类校验。

### 2.2 脚本侧期望输出（概念）

- 成功：返回标准化实体信息，至少包括：
  - `secuCode`：股票代码  
  - `marketChar`：市场代码（如 SH / SZ / BJ / HK / US 等，以接口为准）  
  - `classCode`：市场分类码（用于判断是否属于支持的市场）  
  - `secuName`：证券简称（如有）
- 多个候选时：**取第一个**有效实体。

### 2.3 编排层分支

| 情况 | 处理 |
|------|------|
| 识别成功且属于支持市场 | 进入第二步 |
| 识别结果为空或不支持的市场 | 提示：**「目前仅支持沪深京港美实体进行业绩点评。您提供的实体不在支持范围内，请确认后重新输入。」**（或等价表述），终止流程 |
| 接口失败 / 脚本异常 | 提示接口或系统错误，终止流程 |

---

## 3. 第二步：报告期列表与匹配

### 3.1 报告期列表获取

- 调用 `scripts/normalize_report_period.py`，传入第一步得到的实体信息，由脚本请求**报告期列表接口**，得到该标的**已披露**的报告期集合。

### 3.2 谁来选择具体报告期（核心）

- **由上层模型（或调用方）**根据用户问句、实体及完整报告期列表，选定一个 `reportDate`（格式一般为 `YYYY-MM-DD`）。这是分步骤编排中的关键决策步骤。
- 模型选定的 `reportDate` 可通过 `normalize_report_period.py --selected-report-date` 做有效性校验；未传时默认取候选列表第一项（通常为最新）。

### 3.3 用户意图解析（编排层建议）

1. 从问句中解析报告期意图：年份、季度类型、是否「最新」「最近」等。  
2. 结合市场类型在列表中匹配**最佳**报告期。  
3. 若用户**完全未指定**报告期，或仅表述为「最新」「最近一期」等，通常对应列表中的**最新一期**（列表排序以接口返回为准，一般为新→旧）。

### 3.4 A 股报告期匹配（自然年≈财年，可直接按日期匹配）

- 用户指定**年份 + 季度**（如「2024 年三季报」）→ 在列表中**精确匹配**对应报告期末日。  
- 用户只指定**季度**未指定年份（如「三季报」）→ 在列表中匹配**最近一次**出现的该季度。  
- 用户只指定**年份**未指定季度（如「2024 年」）→ 匹配该自然年内列表中**最新可用**的一期。  
- 季度与报告期常见对应：一季报 / 半年报 / 三季报 / 年报（全年）等，以列表中实际 `reportDate` 为准。

### 3.5 港美股报告期匹配（财年与自然年可能不一致）

- 港美股公司财年结束日可能不是 12 月；用户说的「2024 年年报」在不同公司上对应的**自然日期区间**可能不同。  
- 建议规则：  
  - 用户使用**自然年**表述 → 按自然年维度在列表中匹配；  
  - 用户使用**财年**表述（如 FY2024）→ 按财年语义匹配；  
  - 若财年与自然年表述**产生歧义**，优先按**财年**匹配，并在最终输出中**说明**该公司财年结束月份及本次匹配到的自然报告期。

### 3.6 匹配失败与近似匹配

| 情况 | 编排层建议 |
|------|------------|
| 列表中**无精确匹配** | 可选用**时间最接近**的一期，并明确提示：「未找到精确匹配的报告期，已为您匹配最接近的报告期：{实际 reportDate }」 |
| 报告期列表**为空** | 提示「暂无该实体的可用报告期数据」，终止 |
| 接口失败 | 提示「暂时无法获取报告期信息，请稍后重试」，终止 |

### 3.7 与当前脚本行为的关系（重要）

- `choose_report_option_by_model(..., strict=True)` 下：若传入的 `selected_report_date` **不在**接口返回的候选列表中，脚本会**报错**，而**不会**静默回退到「最接近一期」。  
- 因此：编排层若采用「最接近报告期」策略，应自行算好目标日期并保证传入值**落在列表中**，或调整调用方式（例如先不向脚本传错误日期）。  
- 未传 `selected_report_date` 时，当前实现为取列表**第一项**作为默认（通常需保证接口列表已按新→旧排序）。

---

## 4. 第三步：业绩点评生成与交付形态

### 4.1 调用方式

- 调用 `scripts/call_review_api.py`，传入实体信息与最终 `reportDate`。
- 日志口径统一：`JSON` 过程日志（如 `01~05.json`、`03_comment_raw.json`）仅在 `debug` 模式下落盘；非 `debug` 模式不生成这些日志文件。
- 附件口径统一：无论是否 `debug`，只要接口返回了 `pdfBase64` / `wordBase64`（及可选的数据底稿 base64），都应正常保存到本地附件目录。

### 4.2 若第二步存在附带说明

- 若报告期为**近似匹配**、或港美股存在**财年说明**，应在最终输出**末尾**一并展示，避免用户误解所选报告期。

- 行间公式：`\[...\]`。  
- 行间公式块内部内容保持与源一致。

## 6. 错误处理汇总（完整表）

| 错误场景 | 处理方式 |
|----------|----------|
| 问句中无实体 | 向用户确认需要点评的公司名称 |
| 实体识别失败 | 提示用户确认公司名称或股票代码 |
| 实体不在支持市场 | 返回「目前仅支持沪深京港美实体进行业绩点评」 |
| 报告期列表为空 | 提示「暂无该实体的可用报告期数据」 |
| 报告期接口调用失败 | 提示「暂时无法获取报告期信息，请稍后重试」 |
| 报告期无法从问句解析 | 引导用户明确报告期（如「请问您想查看哪个季度的报告？」） |
| 报告期在列表中无法精确匹配 | 采用最接近一期并说明；**注意与脚本 `strict` 传参一致** |
| 港美股财年歧义 | 说明财年结束月份及实际匹配到的自然报告期 |
| 业绩点评接口调用失败 | 提示「业绩点评生成失败，请稍后重试」 |
| 用户要求阅读完整正文 | 引导下载附件：「完整正文内容请查看附件中的 PDF 或 Word 版报告」 |
